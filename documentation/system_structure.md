# Data Bundle Reseller Platform – System Architecture & Integration Documentation

## Overview

This document describes the full system architecture and workflow for building a **data bundle reseller platform** that integrates:

* A **customer-facing platform**
* **Paystack payment processing**
* **GHDataConnect API** for bundle delivery
* A **backend order management system**

The platform allows customers to purchase mobile data bundles. Payments are processed through Paystack, and once payment is confirmed, the backend calls the GHDataConnect API to deliver the bundle while deducting credit from the reseller’s wallet.

---

# 1. System Roles

## 1. Customer

The end user purchasing a data bundle.

Responsibilities:

* Select bundle
* Enter phone number
* Make payment

---

## 2. Your Platform

Your website or mobile application.

Responsibilities:

* Display bundles and prices
* Initiate payments
* Receive payment confirmations
* Send purchase requests to backend

---

## 3. Payment Gateway (Paystack)

Responsibilities:

* Process payments
* Send payment confirmation via webhook
* Transfer funds to your Paystack account

Important Rule:

Bundle delivery **must only occur after Paystack confirms payment success**.

---

## 4. Backend Core System

The backend is responsible for:

* Creating payment transactions
* Handling Paystack webhook confirmations
* Calling the GHDataConnect API
* Managing orders
* Checking delivery status
* Logging transactions

Recommended backend frameworks:

* FastAPI (Python)
* Node.js (Express / NestJS)

---

## 5. GHDataConnect System

External API that:

* Holds reseller wallet balance
* Processes bundle purchases
* Delivers data bundles to customers

All bundle purchases **deduct from your wallet balance.**

---

# 4. Payment Initialization

Your backend creates a Paystack payment transaction.

Example flow:

```
POST /initialize-payment
```

Request body:

```
{
  "email": "customer@email.com",
  "amount": 600,
  "reference": "ORDER_12345"
}
```

Customer completes payment on Paystack checkout.

---

# 5. Paystack Webhook Confirmation

Paystack sends webhook to your backend when payment succeeds.

Example endpoint:

```
POST /webhooks/paystack
```

Webhook event:

```
event: charge.success
```

Backend must verify:

```
payment_status == success
```

Only then proceed to bundle purchase.

---

# 6. Create Bundle Order (GHDataConnect API)

Endpoint used to deliver the bundle.

```
POST /v1/createIshareBundleOrder
```

Example request:

```
{
  "reference": "1732323325876",
  "msisdn": "0276895857",
  "capacity": 1000
}
```

Parameters:

| Field     | Description                              |
| --------- | ---------------------------------------- |
| reference | Unique order ID generated by your system |
| msisdn    | Customer phone number                    |
| capacity  | Bundle size in MB                        |

Example values:

| Capacity | Bundle |
| -------- | ------ |
| 1000     | 1GB    |
| 2000     | 2GB    |

---

# 7. Wallet Balance Management

Before sending bundles, the backend can check wallet balance.

Endpoint:

```
GET /v1/getWalletBalance
```

Example request:

```
curl --location '/v1/getWalletBalance'
```

Example response:

```
{
 "success": true,
 "data": {
   "balance": "207.46"
 }
}
```

Meaning:

Your reseller wallet contains **207.46 credits**.

Each order deducts from this wallet.

---

# 8. Order Status Verification

To confirm delivery status.

Endpoint:

```
GET /v1/checkOrderStatus/:reference
```

Example request:

```
curl --location '/v1/checkOrderStatus/173232332595467876'
```

Example response:

```
{
  "success": true,
  "data": {
    "order_id": 16,
    "reference": "173232332595467876",
    "status": "pending",
    "recipient": "0591178627",
    "network": "MTN",
    "capacity": 1,
    "price": 4.1,
    "payment_status": "completed",
    "created_at": "2025-12-21T18:54:57+00:00",
    "delivered_at": null
  }
}
```

Possible statuses:

| Status    | Meaning                 |
| --------- | ----------------------- |
| pending   | Bundle still processing |
| completed | Bundle delivered        |
| failed    | Delivery failed         |

# 9. Bundle Discovery

To fetch available bundles.

Endpoint:

```
GET /v1/getAllNetworks
```

Example response:

```
{
  "success": true,
  "data": [
    {
      "name": "MTN",
      "key": "mtn",
      "bundles": [
        {
          "id": 1,
          "capacity": 1,
          "price": 4.1
        }
      ]
    }
  ]
}
```

Fields:

| Field    | Meaning               |
| -------- | --------------------- |
| name     | Network name          |
| key      | Network identifier    |
| bundles  | Available bundle list |
| capacity | Bundle size           |
| price    | API cost per bundle   |

Your platform can sell bundles at **higher prices for profit**.

# 12. Important Security Rules

## Unique Order References

Every order must have a unique reference.

Example:

```
ORDER_20260313_001
```

or

```
UUID
```

---

## Webhook Verification

Always verify Paystack webhook signatures to prevent fraud.

---

## Prevent Duplicate Delivery

Webhook events may retry.

Your system must ensure:

```
if order_already_processed:
    ignore_request
```
